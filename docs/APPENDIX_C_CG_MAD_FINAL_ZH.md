# 附录 C  CG-MAD 的实现细节：Agent 实例化、信息包、路径表示与提示词模板

本附录给出 CG-MAD（Causal Graph-augmented Multi-Agent Debate）的关键实现细节，以提升方法的可复现性与可审计性。内容包括：  
（i）三类 Agent 的实例化方式与裁判隔离策略；  
（ii）统一信息包（evidence pack）的字段定义与构建流程；  
（iii）“路径信息”的结构化表示方式；  
（iv）路径层与候选位置层辩论任务的输入/输出 schema；  
（v）支持者、反对者与裁判三类 Agent 的提示词模板；  
（vi）结构化解析与异常处理策略；  
（vii）与主文第 3.1.2 节中各统计量和更新公式的对应关系。

## C.1 Agent 实例化与裁判隔离

### 同模型、角色化实例化

CG-MAD 包含支持者（Proponent）、反对者（Skeptic）与裁判（Judge）三类 Agent。三者共享统一的输入组织方式，但通过角色条件化提示词（role-conditioned prompts）完成职能分工，而非依赖不同的自由对话历史或外部 ensemble。该设计有助于降低模型差异带来的额外干扰，使性能变化主要归因于辩论机制与信息组织方式本身。

在实现上，系统保留三类可独立配置的模型接口：  
（1）`llm`：支持者与反对者默认共享的底座模型；  
（2）`judge_llm`：裁判模型，可与 `llm` 相同，也可单独替换为推理能力更强的模型；  
（3）`summary_llm`：用于路径节点语义摘要生成的模型，可与辩论模型共享，也可单独配置。  

因此，从方法设计上看，Judge 使用更强模型是被显式支持的；但当未额外指定时，系统仍可退化为三类 Agent 共享同一底座模型的配置。该设计兼顾了实现简洁性与后续可扩展性。

### 固定模板与结构化输出

三类 Agent 均采用固定的系统提示与用户提示模板，并被要求输出预定义 JSON schema。固定模板与结构化输出具有两方面作用：  
（1）便于自动聚合路径层与候选位置层的胜率信号，并将其反馈至边权更新与候选位置排序；  
（2）降低 prompt 变体对辩论结果的敏感性，从而提升不同重复运行之间的可比性。

在实现中，所有 Agent 输出均首先经过结构化解析；若模型返回内容不可解析，则进入固定 fallback 分支，以保证后续统计与更新流程始终可执行。

### 裁判隔离策略（Blind Judging）

为减弱自由文本辩论中的语言偏置与提示注入风险，Judge 不直接读取支持方与反对方的长对话过程，而仅接收以下三类输入：  
（1）同一份缺陷描述；  
（2）同一份路径证据包或候选位置证据包；  
（3）支持方与反对方的结构化输出。  

Judge 在裁决时优先考察：  
（i）failure anchor 对齐程度；  
（ii）解释链是否连贯；  
（iii）支持方提出的机制链是否成立；  
（iv）反对方指出的语义断裂、弱边或噪声节点是否成立；  
（v）当前候选位置是否具备可干预性（actionability）。  

由于输入中不包含情绪化措辞或“自信程度”式修辞，Judge 的裁决更依赖共享证据包中的显式证据字段，而非自由文本说服力。

### 可控随机性

在重复运行中，系统仅改变随机种子以驱动采样与判决分支；其余配置保持一致，包括候选位置集合、CRG 路径构建规则、evidence pack 字段、提示词模板、最大轮数、早停条件以及上下文裁剪参数。由此，重复实验之间的差异主要反映采样与辩论机制引入的随机性，而非输入构造方式的漂移。

## C.2 统一信息包（Evidence Pack）：字段定义与构建

对于每个 issue，CG-MAD 将输入统一组织为一份 evidence pack，供 Support、Oppose 与 Judge 三类 Agent 共享。当前实现中的 evidence pack 主要包含两部分：缺陷描述、失败锚点与候选位置上下文。

### 缺陷描述

- `issue_text`：来自数据集的原始问题描述（issue statement / problem statement），通常包含失败现象、报错信息以及复现线索等。

### 失败锚点

系统以 `FailureAnchorPack` 表示失败入口锚点，每个锚点包含如下字段：

| 字段 | 含义 |
| --- | --- |
| `anchor_id` | 唯一锚点标识，供路径与裁判显式引用 |
| `symptom_type` | 失败证据类型，如 failing test file 或 problem statement |
| `symptom_location` | 失败观测位置，如测试文件路径或症状对应位置 |
| `symptom_message` | 失败症状文本或关键错误信息 |
| `evidence_refs` | 与该锚点相关的可引用证据字段 |

失败锚点承担了“路径末端对齐目标”的作用，使后续 Support、Oppose 与 Judge 均能围绕同一失败入口展开证据比较，而不是在抽象意义上讨论“是否相关”。

### 候选位置上下文

系统以 `CandidateContext` 表示候选位置及其局部代码上下文，每个候选位置包含如下字段：

| 字段 | 含义 |
| --- | --- |
| `candidate_id` | 候选代码实体的唯一标识 |
| `display_id` | 用于提示词展示的可读标识 |
| `entity_type` | 实体类型，如 `function`、`class` 等 |
| `file_path` | 源代码文件路径 |
| `code_snippet` | 候选位置所在函数或局部代码窗口的上下文片段 |

因此，CG-MAD 输入中的“候选位置”并非仅由符号标识构成，而是同时伴随候选位置对应的局部代码片段。该设计保证三类 Agent 在辩论时既能够引用结构化路径证据，也能够结合候选位置局部上下文理解其可干预性。

## C.3 路径信息的结构化表示（Path Evidence Representation）

为将第 3.1.1 节构建的 CRG 信息输入到 CG-MAD 中，系统将每条候选路径压缩为一个结构化路径证据包（path evidence pack）。路径按照“从候选位置到失败入口”的节点序列组织，并保留节点、边权与 failure anchor 三类证据。

### 节点级字段

当前每个路径节点由 `PathNodeSummary` 表示，包含如下字段：

| 字段 | 含义 |
| --- | --- |
| `entity_id` | 节点唯一标识 |
| `display_id` | 可读展示标识，通常接近 `file::symbol` |
| `name` | 实体名 |
| `entity_type` | 实体类型，如 `function/class/variable/import` |
| `file_path` | 源代码文件路径 |
| `semantic_summary` | 对该节点职责或关键行为的 1–2 句语义摘要 |
| `code_snippet` | 该节点对应的局部代码片段 |
| `edge_weight_to_next` | 与下一节点之间边的当前权重 |

### 路径级字段

当前每条路径对应一个 `PathSummary`，并进一步构造成 path evidence pack，其核心字段包括：

| 字段 | 含义 |
| --- | --- |
| `path_id` | 路径唯一标识 |
| `candidate_id` | 路径所属候选位置 |
| `candidate_context` | 与该路径对应的候选位置上下文 |
| `failure_anchor_refs` | 该路径对齐的失败锚点引用 |
| `nodes` | 路径节点序列 |
| `chain_text` | 可读链式摘要 |
| `evidence_refs` | 汇总后的可引用证据，如 `node:*`、`edge:*`、`anchor:*` |

其中，`chain_text` 用于以可读链式形式概括整条解释链，`evidence_refs` 用于支持后续 Agent 在输出中显式引用节点、边与锚点证据。

### 节点语义摘要生成

节点语义摘要优先由代码图阶段已有摘要或 `summary_llm` 自动生成；若模型不可用，则回退到启发式模板摘要。该设计保证系统即使在摘要模型缺失时仍可稳定运行，同时又允许在条件允许时引入更高质量的语义描述，以增强辩论阶段的可读性与可解释性。

### 路径压缩与提示词输入

为控制 token 开销并提升长路径场景下的辩论稳定性，系统在将 path evidence pack 输入 Agent 提示词之前，会进一步执行长度约束与结构压缩，包括：  
（1）限制代码片段长度；  
（2）限制链式摘要长度；  
（3）当路径过长时，仅保留前后两端关键节点，并以占位符标识中间被省略的节点序列。  

因此，实际进入大模型提示词的“路径信息”是保留证据引用能力的压缩版本，而非未经裁剪的全量路径展开。这一设计在保证结构信息不丢失的同时，使长路径实例仍能在可控上下文预算内完成稳定推理。

## C.4 CG-MAD 各 Agent 的输入输出与信号类型

CG-MAD 在实现上包含两类辩论任务：  
（1）路径层辩论：在同一候选位置的多条路径之间进行比较，用于估计路径层胜率 $P_{path}$；  
（2）候选位置层辩论：在不同候选位置的代表路径之间进行比较，用于估计位置层胜率 $P_{loc}$。  

二者共享相同的角色化设计与基本输出结构，但 `winner` 字段的取值范围会随任务类型而变化。

### 各 Agent 输入输出

| Agent | 输入（字段级） | 输出（JSON schema） | 核心关注点 |
| --- | --- | --- | --- |
| Support（Proponent） | `issue_text` + `path_evidence_pack`（或一对待比较路径） + `failure_anchors` | `claim`, `evidence_refs`, `mechanism` | failure anchor 对齐、关键节点与高权边支撑、解释链连贯性 |
| Oppose（Skeptic） | `issue_text` + `path_evidence_pack`（或一对待比较路径） + `failure_anchors` | `counter_claim`, `weak_links`, `noise_flags` | 语义断裂、弱边依赖、无关职责节点、噪声环节识别 |
| Judge（Evaluator） | `issue_text` + 同一份路径/候选证据包 + 支持/反对双方结构化输出 | `winner`, `key_reasons`, `evidence_refs` | 证据覆盖度、解释链连贯性、反驳是否成立、可干预性 |

其中，`evidence_refs` 被要求显式引用路径中的 `node_id`、边段 `edge:src->dst` 以及 `anchor_id`，以提升辩论过程的可审计性，并抑制仅凭语言风格取得优势的情况。

### `winner` 字段的取值约定

在实现中，`winner` 的取值随辩论任务而变化：

- 对于单路径自辩论，`winner` 取 `support` 或 `oppose`；
- 对于路径对路径比较，以及候选位置对候选位置比较，`winner` 取 `A` 或 `B`。

这一设计使单路径场景与成对比较场景能够共享统一裁判 schema，同时又不损失输出可解析性。

## C.5 提示词模板（Prompt Templates）

当前实现针对路径层、单路径层与候选位置层分别构造固定 prompt，但总体遵循四项一致原则：  
（1）角色说明固定；  
（2）输入字段固定；  
（3）输出 schema 固定；  
（4）证据引用格式固定为 `node:<id>` / `edge:<src->dst>` / `anchor:<id>`。  

下面给出与当前实现一致、可直接用于论文附录展示的模板抽象形式。

### Support Agent 提示词模板

你是自动化程序修复系统中的支持者（Proponent）。给定缺陷描述与一条候选路径（或两条待比较路径）的解释链摘要，你的任务是论证目标路径或目标候选为何更可能解释当前失败。请指出关键节点与高权边如何形成机制链，并显式引用 `node_id`、`edge` 或 `failure_anchor` 作为证据。不得编造未提供的信息。输出必须为严格 JSON。

缺陷描述：  
`{{issue_text}}`

路径信息（path evidence pack）：  
`{{path_evidence_pack}}`

任务：支持该路径或该候选更可能解释失败，并给出机制说明与证据引用（引用 `node_id` / `edge` / `anchor`）。

输出 JSON：

```json
{
  "claim": "...",
  "evidence_refs": ["node:3", "edge:3->4", "anchor:..."],
  "mechanism": "..."
}
```

### Oppose Agent 提示词模板

你是自动化程序修复系统中的质疑者（Skeptic）。给定缺陷描述与一条候选路径（或两条待比较路径）的解释链摘要，你的任务是指出目标路径解释失败的薄弱环节或噪声，例如语义断裂、弱边依赖、包含无关职责节点等。必须显式引用 `node_id`、`edge` 或 `failure_anchor` 作为证据，并给出削弱或否决理由。不得编造未提供的信息。输出必须为严格 JSON。

缺陷描述：  
`{{issue_text}}`

路径信息（path evidence pack）：  
`{{path_evidence_pack}}`

任务：反对该路径或该候选解释失败，指出薄弱节点、薄弱边或噪声环节，并给出理由（引用 `node_id` / `edge` / `anchor`）。

输出 JSON：

```json
{
  "counter_claim": "...",
  "weak_links": ["edge:2->3", "node:4"],
  "noise_flags": ["semantic_gap", "weak_edge_dependency"]
}
```

### Judge Agent 提示词模板

你是自动化程序修复系统中的裁判（Evaluator）。你将看到同一缺陷描述、同一份路径信息或候选位置信息，以及支持方与反对方的结构化要点。你的任务是基于证据做出裁决：优先考察 failure anchor 对齐程度、解释链连贯性、支持方机制是否成立、反对方指出的断裂是否成立，以及当前候选是否具备可干预性。必须引用路径节点、边或锚点作为证据。不得偏袒任何一方。输出必须为严格 JSON。

缺陷描述：  
`{{issue_text}}`

路径信息（path evidence pack）：  
`{{path_evidence_pack}}`

Support（结构化）：  
`{{support_json}}`

Oppose（结构化）：  
`{{oppose_json}}`

任务：裁决哪一方更可信，并给出关键依据与证据引用。

输出 JSON：

```json
{
  "winner": "support|oppose|A|B",
  "key_reasons": ["..."],
  "evidence_refs": ["node:3", "edge:3->4", "anchor:..."]
}
```

## C.6 结构化解析与异常处理策略

为降低提示词变体、格式漂移与自由文本输出带来的不稳定性，CG-MAD 固定所有 prompt 模板与 JSON schema，并对模型输出进行统一的结构化解析。若模型输出不可解析，系统采用两层保护机制：

1. 底层 LLM 接口负责请求级超时控制与重试；
2. CG-MAD 侧在解析失败时回退到固定的 fallback JSON，以保证后续胜率统计、边权更新与候选评估流程不会中断。

因此，更准确的实现性表述是：CG-MAD 采用“固定 schema + 结构化解析 + fallback”的鲁棒执行策略，而非依赖自由文本辩论后的人工后处理。

## C.7 与主文第 3.1.2 节公式的衔接

当前实现与主文第 3.1.2 节中的统计量和更新规则一一对应：

- 路径层辩论结果聚合为 `path_win_rates`，对应主文中的路径层胜率 $P_{path}$；
- 候选位置层辩论结果聚合为 `location_win_rates`，对应主文中的位置层胜率 $P_{loc}$；
- 边权更新遵循加权融合形式

$$
c^{*}=\eta_{1}c^{(0)}+\eta_{2}P_{path}+\eta_{3}P_{loc},
$$

其中 $\eta_{1}+\eta_{2}+\eta_{3}=1$；

- 路径可信度由更新后边权的聚合与长度惩罚共同决定；
- 每个候选位置最终选取在更新边权下可信度最高的代表路径，并以其可信度作为候选位置的最终评估分数。

此外，Judge 输出的结构化理由会被附加到候选评估结果中，用于支撑后续的解释、审计与定性分析。由此，CG-MAD 不仅输出排序结果，也输出可追溯的“为何该路径/该位置更可信”的结构化证据。

## C.8 小结

综上，CG-MAD 的实现遵循“统一信息包输入、角色化结构辩论、裁判隔离、结构化解析、统计信号回写”的总体原则。其核心并非让多个 Agent 展开开放式自由对话，而是在共享的路径证据包之上，以可解析、可引用、可审计的方式逐步估计路径层与候选位置层的可信度，并将这些信号反馈至 CRG 边权与候选排序中。该设计使 CG-MAD 在保持结构先验与解释性的同时，能够在有限上下文预算内稳定运行，并为后续 SRCD 与 TSPF 阶段提供更可靠的因果约束输入。
