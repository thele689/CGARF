# 附录 D  SRCD 的实现细节：输入组织、自反思评审与一致性蒸馏模板

本附录给出 SRCD（Self-Reflection Consistency Distillation）的关键实现细节，以提升方法的可复现性与可审计性。内容包括：  
（i）补丁生成阶段的输入组织方式与动态资源分配；  
（ii）自我反思阶段三类评审维度的具体含义、输出 schema 以及温度调节策略；  
（iii）一致性蒸馏阶段“共识修复模式”的构建方式与一致性打分流程；  
（iv）相关提示词模板；  
（v）结构化解析、重试与 fallback 机制；  
（vi）与主文第 3.2 节公式的对应关系。

## D.1 初始补丁生成：输入组织、动态资源分配与提示词模板

### 条件化输入组织

对于每个候选位置，SRCD 将补丁生成输入统一组织为一份条件化信息包。当前实现中的每个候选输入由 `SRCDCandidateInput` 表示，包含如下字段：

| 字段 | 含义 |
| --- | --- |
| `candidate_id` | 候选位置唯一标识 |
| `candidate_location` | 候选位置的可读展示形式 |
| `file_path` | 源文件路径 |
| `entity_type` | 候选实体类型 |
| `code_context` | 候选位置所在函数或局部窗口的代码片段 |
| `representative_path_id` | CG-MAD 选出的代表路径标识 |
| `representative_path_summary` | 代表路径的压缩链式摘要 |
| `representative_path_evidence` | 代表路径的结构化证据包 |
| `candidate_credibility` | 由 CG-MAD 输出的位置可信度 |
| `normalized_weight` | 动态预算分配后的归一化权重 |
| `allocated_samples` | 分配到该候选位置的总采样次数 |

因此，SRCD 的输入并非仅由“缺陷描述 + 局部代码”组成，而是显式融合了来自 CG-MAD 的结构化因果约束，包括代表路径摘要、代表路径证据以及候选位置可信度。

### 基于候选可信度的动态资源分配

在初始补丁生成之前，SRCD 首先根据 CG-MAD 输出的候选位置可信度进行动态资源分配。设总采样预算为 $N$，候选位置 $l_i$ 的可信度为 $\mathrm{Cred}(l_i)$，则其归一化采样权重定义为：

$$
w_i=\frac{\mathrm{Cred}(l_i)}{\sum_j \mathrm{Cred}(l_j)}.
$$

在实现上，系统先根据 $w_i \cdot N$ 计算各候选位置的原始分配值，再采用最大余数法（largest-remainder rounding）将其离散化为整数采样预算，以保证各候选位置的采样数之和严格等于总预算 $N$。当 $N \ge K$（其中 $K$ 为候选位置数）时，系统额外保证每个候选位置至少保留 1 个初始采样槽位，从而满足“每个候选位置先生成一个初始补丁”的设计要求。

### 初始补丁生成流程

在完成预算分配后，系统首先仅为每个候选位置生成一个初始补丁，作为后续自我反思与补丁变体扩展的起点。初始补丁的生成遵循以下原则：

1. 修改应尽量与代表路径刻画的致因机制一致；
2. 修改范围应保持局部、最小；
3. 应尽量保持与缺陷无关的原有语义、接口与控制流不变；
4. 输出必须采用严格的 Search/Replace 格式。

需要指出的是，当前实现不仅要求输出为 Search/Replace 格式，还额外施加了若干可执行性约束，包括：

- `SEARCH` 与 `REPLACE` 不得完全相同，即不得产生 no-op patch；
- 应优先修改最小连续代码片段，而非默认重写整函数；
- 禁止以宽泛的 `try/except Exception` 包裹整个函数作为退化式修复；
- 不鼓励与问题无关的重构、重命名或大范围逻辑改写。

若初始生成结果无效、过宽或不产生真实修改，系统会进一步触发更严格的重试提示与强制局部编辑提示；若多轮 LLM 生成仍无法得到有效补丁，则回退到确定性的模板/变异策略，以保证流程的可执行性。

### 初始补丁生成提示词模板

下面给出与当前实现一致的初始补丁生成模板抽象形式。

你是自动化程序修复系统中的补丁生成代理。给定缺陷描述、一个候选修复位置的代码片段，以及从该位置到失败入口的代表路径摘要，你的任务是在尽量局部修改的前提下，为该候选位置生成一个初始补丁。补丁应满足以下要求：

1. 修改应尽量与代表路径所刻画的致因机制一致；
2. 不要引入与问题无关的大范围改动；
3. 尽量保持原有代码语义、接口行为与无关控制流不变；
4. 不要使用宽泛的整函数 `try/except` 包裹作为退化修复；
5. 输出必须采用严格 Search/Replace 格式，不要输出解释性文字或额外说明；
6. `SEARCH` 与 `REPLACE` 必须至少在一条可执行语句上不同。

`{{issue_text}}`

候选位置：  
`{{candidate_location}}`

候选位置代码片段：  
`{{code_context}}`

代表路径摘要：  
`{{representative_path}}`

生成要求：

- 尽量只修改与该缺陷直接相关的语句；
- 修改应与代表路径中的关键节点/关键机制保持一致；
- 避免无关重构、重命名或大范围逻辑调整；
- 优先返回最小连续片段的 Search/Replace 补丁。

输出格式要求：

```text
<<< SEARCH
...
===
...
>>> REPLACE
```

## D.2 自我反思评审：输入组织、评分维度与提示词模板

### 评审输入组织

在自我反思阶段，SRCD 对当前补丁进行一次结构化评审，以判断该补丁是否既能够针对目标缺陷进行有效修复，又尽量保持原有程序语义不发生与缺陷无关的漂移。当前实现中，反思输入由以下信息共同构成：

1. `issue_text`：缺陷描述；
2. `candidate_location`：候选位置；
3. `code_context`：该候选位置对应的局部代码片段；
4. `representative_path_summary`：由 CG-MAD 输出的代表路径摘要；
5. `patch_text`：当前待评审的 Search/Replace 补丁。

值得强调的是，SRCD 的反思对象是“当前补丁本身”，而不是抽象地判断该候选位置是否值得修复；因此三类评审维度均围绕补丁自身展开。

### 三类评审维度

#### （1）语义一致性（Semantic Consistency）

语义一致性用于判断补丁在修复目标缺陷的同时，是否尽量保持与该缺陷无关的原有程序语义不变。评审时重点检查：

1. 是否改变了当前函数或模块原本的职责；
2. 是否改变了输入输出关系、返回值语义或接口行为；
3. 是否引入了新的副作用，或改变了原有状态更新逻辑；
4. 是否改变了异常处理路径、边界条件判断或与缺陷无关的控制流；
5. 是否修改了与当前失败现象无直接关联的代码语义。

#### （2）因果链匹配（Causal Alignment）

因果链匹配用于判断补丁是否与代表路径刻画的致因机制一致。评审时重点检查：

1. 补丁修改的位置是否位于代表路径的关键节点或关键连接附近；
2. 修改是否能够解释“为何该改动可以缓解失败现象”；
3. 补丁是否围绕路径中与失败入口相关的变量、条件、调用或异常分支展开。

#### （3）修改最小化（Minimal Edit）

修改最小化用于判断补丁是否将修改限制在必要范围内。评审时重点检查：

1. 是否避免了与问题无关的大范围代码改动；
2. 是否避免了不必要的重构、重命名或新增逻辑分支；
3. 是否将修改限制在最小作用域或最小连续代码片段内。

### 结构化输出与分值映射

模型需对三个维度分别给出五级等级判断：

- `fully_not`
- `partially_not`
- `neutral`
- `partially_yes`
- `fully_yes`

当前实现将其映射为稳定的数值分数：

| 等级 | 分值 |
| --- | --- |
| `fully_not` | 0.0 |
| `partially_not` | 0.25 |
| `neutral` | 0.5 |
| `partially_yes` | 0.75 |
| `fully_yes` | 1.0 |

三个维度的分数按加权和聚合为反思分数 `combined_reflection`。当前默认权重为等权配置，即三维度各占 $1/3$，但实现上保留权重可配置入口。除分数外，模型还需给出：

- 每个维度对应的理由；
- 一条 `revision_suggestion`，用于指导下一轮补丁生成；
- 基于反思结果自动计算得到的 `suggested_temperature`，用于调节后续补丁生成温度。

### 温度调节与后续补丁生成

当前实现中，反思分数除用于质量评估外，还用于驱动后续生成的采样温度调整：

- 当反思分数高于阈值时，降低生成温度，以增强稳定性；
- 当反思分数低于阈值时，提高生成温度，以增强探索性。

更重要的是，后续迭代补丁并不是“在前一个补丁上继续局部编辑”，而是基于以下输入重新生成一个新的候选补丁：

1. 原始缺陷描述；
2. 相同候选位置的局部代码上下文；
3. 相同代表路径摘要；
4. 当前轮反思得到的结构化评审结果与 `revision_suggestion`。

换言之，反思阶段提供的是“对上一轮补丁的结构化反馈”，而后续生成阶段执行的是“在相同候选位置上基于反馈重新生成一个新的补丁”。

## D.3 共识修复模式提取：输入组织与提示词模板

### 输入组织

在一致性蒸馏阶段，SRCD 不直接比较补丁文本表面的相似性，而是先对同一候选位置下生成的一组补丁进行归纳，总结它们共同体现的修复意图，从而形成“共识修复模式”（consensus repair pattern）。当前实现中，该阶段的输入包括：

1. 候选位置标识；
2. 候选位置代码片段；
3. 该候选位置下生成的一组补丁及其轮次信息。

模型的任务不是逐条复述每个补丁，而是抽取其中反复出现的修改规律，例如这些补丁是否都围绕同一变量、同一条件判断、同一 API 调用或同一异常分支展开修改，以及这些修改共同试图修正的机制是什么。

### 共识模式的结构化表示

当前实现中，共识修复模式由 `ConsensusPatternSummary` 表示，包含如下字段：

| 字段 | 含义 |
| --- | --- |
| `candidate_id` | 候选位置标识 |
| `candidate_location` | 候选位置可读表示 |
| `shared_edit_intent` | 多数补丁共同体现的修改意图 |
| `shared_target_entities` | 多数补丁共同作用的目标实体 |
| `shared_mechanism` | 多数补丁共同试图修正的机制 |
| `shared_constraints` | 多数补丁共同遵循的局部修改约束 |
| `source` | 共识模式来源（LLM 或启发式） |

### 提示词模板

你是自动化程序修复系统中的归纳代理。给定同一候选位置下的一组候选补丁，请总结这些补丁中反复出现的修改模式，并提炼出它们共同体现的修复意图。不要逐条复述每个补丁，而要归纳它们在以下方面的共性：

1. 主要修改对象是什么（如变量、条件、调用、异常分支、状态更新语句等）；
2. 主要修改方式是什么（如增加条件判断、调整分支逻辑、修正参数、补充状态检查等）；
3. 这些修改共同试图修正的机制是什么；
4. 它们在修改时共同遵循了哪些局部约束（如局部修改、避免重构、保持接口不变等）。

请输出严格 JSON，不要包含额外解释性文字。

候选位置：  
`{{candidate_location}}`

候选位置代码片段：  
`{{code_context}}`

候选补丁集合：  
`{{patch_list}}`

输出 JSON：

```json
{
  "shared_edit_intent": "...",
  "shared_target_entities": ["..."],
  "shared_mechanism": "...",
  "shared_constraints": ["..."]
}
```

### LLM 与启发式双通道

当前实现优先尝试通过 LLM 抽取共识修复模式；若 LLM 调用失败或输出不可解析，则回退到启发式归纳策略。启发式策略通过分析 Search/Replace 补丁中的标识符、条件语句、返回语句等线索，近似恢复补丁集合的共同修改意图与局部约束，从而保证 3.2.3 阶段在无 LLM 或 LLM 不稳定场景下仍可运行。

## D.4 一致性分数与蒸馏分数的计算实现

### 补丁嵌入表示

在计算补丁与共识模式之间的一致性时，当前实现并不直接对完整补丁文本进行嵌入，而是优先提取 Search/Replace 中真正发生变化的行，并构造成紧凑的“编辑文本表示”（embedding text），形式上接近：

- Removed lines: ...
- Added lines: ...

若补丁不存在有效的 Search/Replace 结构，则退化为直接使用补丁原文。该设计能够减少大量重复上下文对嵌入相似度的干扰，使一致性分数更聚焦于“补丁真正改了什么”。

### 候选内共识中心与分散惩罚

对同一候选位置下的一组补丁，系统首先计算：

1. 共识模式文本的嵌入向量；
2. 各补丁编辑文本的嵌入向量；
3. 同一候选位置下所有补丁嵌入向量的平均向量（即该候选位置的嵌入中心）。

随后，系统为每个补丁计算：

- `consistency_score`：补丁与共识模式文本之间的余弦相似度；
- `embedding_similarity`：补丁与候选位置嵌入中心之间的余弦相似度；
- `dispersion_penalty`：该候选位置内所有补丁相对嵌入中心的平均离散惩罚。

需要说明的是，当前实现中 `dispersion_penalty` 是候选位置级别的统计量，因此同一候选位置下所有补丁共享相同的分散惩罚项；而 `embedding_similarity` 主要作为分析与审计字段保留，并不直接进入最终蒸馏公式。

### 蒸馏分数

对候选位置 $l_i$ 下的补丁 $p_{i,n}$，当前实现中的蒸馏分数写为：

$$
\mathrm{Dist}(p_{i,n})=\mathrm{Ref}(p_{i,n})+\alpha \cdot \mathrm{Cons}(p_{i,n})-\beta \cdot \mathrm{Disp}(l_i),
$$

其中：

- $\mathrm{Ref}(p_{i,n})$ 为第 3.2.2 节得到的反思分数；
- $\mathrm{Cons}(p_{i,n})$ 为补丁与共识修复模式之间的一致性分数；
- $\mathrm{Disp}(l_i)$ 为候选位置内补丁集合的平均分散惩罚；
- $\alpha$ 与 $\beta$ 分别控制共识加权与分散惩罚强度。

当前默认配置中，$\alpha = 0.8$，$\beta = 0.5$。系统随后按蒸馏分数对同一候选位置下的补丁进行排序，并保留 top-$k$ 个补丁作为该候选位置的一致性蒸馏输出。

## D.5 嵌入后端与可运行性保证

为兼顾可复现性与实际部署便利性，当前实现支持两类嵌入后端：

1. 本地嵌入后端：可在 CPU 上加载本地或 HuggingFace 兼容的嵌入模型；
2. API 嵌入后端：可通过 SiliconFlow 的 embedding API 调用 `Qwen/Qwen3-Embedding-0.6B`。

系统会记录：

- 配置的嵌入模型名；
- 实际生效的嵌入模型名；
- 当前嵌入后端模式；
- 模型加载错误信息（若有）。

该设计保证即使某一嵌入后端不可用，系统仍可通过后备路径继续完成一致性蒸馏，并保留足够的元信息供后续审计。

## D.6 结构化解析、重试与 fallback 机制

SRCD 在三个阶段均采用“固定模板 + 结构化解析 + 失败回退”的鲁棒执行策略：

### 3.2.1 初始补丁生成

- 优先使用主提示词生成初始补丁；
- 若结果无效、过宽或不产生真实修改，则使用更严格的 retry prompt；
- 若仍失败，则使用强制局部编辑 prompt；
- 若多轮 LLM 生成仍失败，则回退到确定性的模板/变异策略。

### 3.2.2 自我反思评审

- 优先使用固定 JSON schema 的结构化反思 prompt；
- 若模型输出不可解析，则退化为启发式结构化评审；
- 启发式结构化评审仍输出相同 schema，以保证后续阶段接口一致。

### 3.2.3 一致性蒸馏

- 优先使用 LLM 提取共识修复模式；
- 若输出不可解析或调用失败，则退化为启发式共识提取；
- 无论采用哪种路径，后续一致性打分与排序接口保持一致。

因此，SRCD 的稳定性并不依赖某一次单独的大模型输出是否完美，而是建立在统一 schema、结构化解析与多级 fallback 的整体鲁棒执行设计之上。

## D.7 与主文第 3.2 节公式的衔接

当前实现与主文第 3.2 节中的三个子阶段对应如下：

1. **3.2.1 动态资源分配**：  
   候选位置依据 CG-MAD 输出的可信度进行预算分配，并保证在预算足够时每个候选位置至少生成一个初始补丁。

2. **3.2.2 结构化自反思**：  
   每个补丁围绕语义一致性、因果链匹配与修改最小化三个维度进行结构化评审，并将五级判断映射为稳定数值分数，再聚合得到反思分数 $\mathrm{Ref}(p_{i,n})$。反思结果同时驱动下一轮补丁重新生成时的温度调节。

3. **3.2.3 一致性蒸馏**：  
   同一候选位置下的补丁集合先抽取共识修复模式，再计算补丁与共识模式的一致性分数，并结合候选内分散惩罚形成最终蒸馏分数 $\mathrm{Dist}(p_{i,n})$，最后保留每个候选位置下分数最高的若干补丁。

## D.8 小结

综上，SRCD 的实现遵循“因果约束输入、动态预算分配、结构化反思评审、反馈驱动重新生成、一致性蒸馏筛选”的总体流程。其核心思想并非简单地对候选位置进行多次独立采样，而是在候选位置可信度、代表路径约束与结构化反思反馈的共同作用下，将补丁生成过程从“高随机性的局部试探”转化为“可解释、可控、可审计的迭代优化过程”。这一实现方式既保留了修复生成所需的探索性，又通过反思与蒸馏增强了补丁集合的整体稳定性，为后续 TSPF 阶段提供更高质量的候选补丁输入。
